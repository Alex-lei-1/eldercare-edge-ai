它做什么

两个 AI 工作负载共用一块低功耗板子、一套摄像头 + 麦克风。

视觉

姿态跌倒检测

NPU 上的 YOLOv8n-pose 输出每人 17 个关键点。时序状态机(NORMAL → WARNING → LYING → FALL_CONFIRMED)综合身体角度、离地高度与长宽比随时间判定,确认跌倒后带抓拍报警。

语音

全离线语音陪伴

唤醒词 → 端侧 ASR → 本地 LLM → 神经 TTS,配声学回声消除,助手不会把自己的话当成用户。跌倒后主动问"您还好吗?",老人可随时插话"我没事"/"救命"决定是否升级。

端到端架构

摄像头与 ISP → NPU 推理 → 三路输出 → 跌倒判定 → 消息总线 → 家属报警与语音,外加并行的语音陪伴管线与 systemd 运维层。

跌倒报警链路 · 逐步拆解

从光子到家属通知——每一步都在板上完成。

  1. 采集。 ov13855 → RKAIQ ISP(自动曝光、60 Hz 抗闪、弱光调优)→ V4L2 NV12 DMA-BUF,8 缓冲池。
  2. 推理。 采集线程(A55)把帧交给 3 个并行 RKNN context(A76 + NPU),各自 RGA NV12→RGB 后跑 YOLOv8n-pose 640、后处理与跟踪——~15 fps,零丢帧。
  3. 分发。 关键点走 UDP;带 AI 叠加的 H264 走 RTSP(硬件 MPP 编码,多客户端);原始帧进 tmpfs 的硬件 JPEG 环作为事件证据。
  4. 判定。 跌倒服务把关键点喂进时序状态机,综合身体角度、离地高度、长宽比并维持一段时间后确认跌倒。
  5. 通知。 FALL_CONFIRMED 时发布到 MQTT;报警服务推送家属(ntfy),并附上从 JPEG 环取的抓拍。
  6. 问询。 语音服务非阻塞地说"您还好吗?",允许老人插话,从转录里减去提示音自身的回声,再判 ok / sos 决定安抚还是升级。

工程亮点

真正有意思的决策——靠测量,而非猜测。

640 模型跑出 384 的速度

快的 384 模型在躺姿/远距离会漏检(约 81% vs 640 的 100%)。不牺牲精度,改用三核并行 RKNN context 把 640 拉回 ~15 fps——精度与吞吐兼得。

零拷贝的坑

"理所当然"的完整 rknn_set_io_mem 零拷贝路径,实测对 640 反而慢一倍。保留 RGA 预处理 + rknn_inputs_set 才是赢家——提醒自己:要测,别假设。

工业级弱光 ISP

不靠软件提亮,而是在 IQ 层调 RK3588 硬件 ISP(RKAIQ)——曝光曲线、增益、60 Hz 抗闪——让自动曝光适配暗场且无条带。

听得见、不自言自语

全双工 PipeWire 助手跑 speex AEC + 回声包络门,既不把自己的 TTS 转录进来,又能让真人越过喇叭插话。

可靠的 RTSP

实时流用按客户端独立的 GStreamer media/appsrc 扇出 + 独立时间戳,支持重连、多手机同看。

验证过,不靠假设

部署的 C++ 跌倒逻辑是 Python 参考实现的逐行移植,并在 35 段回放语料上证明逐字节一致——评测分数描述的就是真正上线的代码。
